{% block title %}The Backend{% endblock %}

I have been developing for the web in one form or another since 2006. As the use of JavaScript became more and more powerful in the browser, it also became a language usable on the backend via Node.js. This pervasiveness of JavaScript across the stack means that I have been able to do a lot with just this one language. Having now evolved my career from the browsers to backends, through DevOps and infrastructure, into management, and now back to full-stack development, I thought it was time to learn some new backend languages. This project is focusing on Python.

This site uses the Python-based FastAPI server, and UV manages the build process. While FastAPI--and most other API frameworks are generally used to return JSON data, this one is returning HTML pages, HTML partials, and event streams. It's not _uncommon_ for web servers and APIs to also return static pages, but because of an accident of history, we tend to think of APIs (and specifically REST APIs) as returning only JSON, and the returning of HTML is often treated as a separate, or parallel concern to returning JSON to the client.

Carson Gross, creator of the hypermedia library htmx, writes very well about this in his book Hypermedia Systems, so I won't go into the whole history here, but it is definitely worth a read. The short version of the tale is that REST APIs are specifically supposed to return the "state of the application", but that was intended to refer to the rendered state and the relevant user controls; i.e. markup which included buttons, forms, and links. Very much the inverse of how the industry has come to use the term.

This backend doesn't do that much (yet) but it is set up to be more of a RESTful API in that traditional sense. Even though it is server-rendered, it remains reactive in the sense that the server is able to very quickly send full pages or small snippets of markup that is computationally very inexpensive to push into the DOM--far less expensive than sending data to a client application to be processed and rendered using JavaScript. In fact, by leaning into server-rendered pages and the latest web-standards, we get to rely on highly optimized browser capabilities whose underlying code is written in Rust and C++.

The more time I spend writing for the web in this way, like I did early in my career--but decorated with tools like Datastar (another hypermedia library based on many of the goals and ideas behind htmx), the more I find the server-rendered page to be way more fun to develop. It is lighter and leaner, more effective and scalable, and less complex or expensive than the single-page client applications and HTTP APIs that dominate the industry thinking. HTML Custom Elements, HTML Templates, and the Shadow DOM have been fully supported across browsers for a few years now, and while some struggle with the workflows and conventions of these tools, I have personally found that struggle coming from habitually reaching for client-rendered single-page application model. The more I am able to adhere to the latest standards and practices of the browser, embracing the multi-page application model and server-side rendering, HTML Web Components come to life.

One interesting thing of note that this backend does, is open up an SSE stream for certain responses using Datastar's Python SDK. This allows the server to stream data, stateful markup, or in the case of the story time demo, text content in real-time. This makes the kinds of eventual consistency concerns that come with event-driven systems fairly trivial to address in the browser. This demo project does not directly integrate an event-based workflow, but the Story Time demo should at least illustrate how fast this toolchain can handle incoming events in the browser.
